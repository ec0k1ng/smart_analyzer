# TCS Smart Analyzer 关键决策记录

## D-001 当前用户输入模型收敛为 raw_inputs + 手工接口映射

- 状态：已采纳
- 决策：用户侧只保留 raw_inputs 与接口映射表中的实际信号名，不再让用户理解 signal_aliases 或自动猜列体系。
- 原因：旧别名叙事会让接口映射责任边界混乱，也不利于文档和 GUI 一致。

## D-002 KPI 必须强制提供过程曲线

- 状态：已采纳
- 决策：calculate_kpi_series 是正式 KPI 文件的强约束。
- 原因：曲线页、结果核查和可解释性都依赖 KPI 过程曲线。

## D-003 KPI 的 trend_source 必须等于 KPI 自身 name

- 状态：已采纳
- 决策：禁止把 trend_source 指向临时列名或外部列名。
- 原因：这样可以简化结果页、曲线页和导出链路中的命名一致性。

## D-004 多 KPI 共用中间量必须优先抽成派生量

- 状态：已采纳
- 决策：公共中间量优先进入 derived_signals，而不是复制粘贴进多个 KPI 文件。
- 原因：减少重复逻辑，便于共享和调试。

## D-005 派生量模板必须强制包含 description、algorithm_summary、CALIBRATION

- 状态：已采纳
- 决策：派生量文件不再只是代码片段，必须具备自解释能力。
- 原因：后续接手者和 AI 辅助编辑都需要明确语义和调参边界。

## D-006 派生量水平直线语义必须先确认

- 状态：已采纳
- 决策：如果派生量返回的是全时段恒定值，必须先确认它表达的是文件固有标量属性；未确认前不允许默认采用。
- 原因：水平直线极易掩盖“本应是过程曲线”的理解错误。

## D-007 面向用户的报告导出仅保留 HTML 与 Word

- 状态：已采纳
- 决策：默认导出物聚焦 HTML 与 Word，不再把 Excel/JSON 作为主交付结果。
- 原因：当前用户场景更关心可阅读报告，而不是原始结构化中间件。

## D-008 同一路径文件允许按不同 KPI 分组重复分析

- 状态：已采纳
- 决策：结果唯一键采用“真实文件路径 + KPI 分组”，而不是仅按文件路径去重。
- 原因：同一文件在不同分析组合下需要并存结果与曲线信号。

## D-009 当前规则判定收敛到 KPI 文件，不再以独立 rule_specs 为执行入口

- 状态：已采纳
- 决策：用户可见规则结果由 KPI_DEFINITION 派生。
- 原因：减少双维护、降低接手成本，并让规则和指标语义保持同源。

## D-010 文档必须按代码当前事实维护，不允许只留在聊天记录中

- 状态：已采纳
- 决策：需求、现状、验证、交接、架构、实现逻辑必须全部落盘。
- 原因：本项目已经进入多人可接手阶段，聊天记录不是可靠知识库。
