# TCS Smart Analyzer 当前阶段完整需求说明书

## 1. 产品目标

当前阶段的目标是交付一套本地离线运行的 TCS 数据分析桌面工具，让工程师在不写代码或只做少量 Python 插件维护的前提下，完成以下闭环：

1. 导入原始日志
2. 把数据映射为统一标准信号
3. 计算派生量与 KPI
4. 基于 KPI 给出可解释的判定结果
5. 在 GUI 中查看结果和曲线过程
6. 导出 HTML/Word 汇总报告

## 2. 用户视角必须成立的模型

当前用户只需要理解两层输入概念：

1. KPI/派生量文件中的 raw_inputs 名称
2. 接口映射表中由用户手工填写的实际信号名

下面这些旧概念不应再出现在对外需求描述中：

- signal_aliases
- 自动猜原始列名
- feature_inputs 黑盒公共预处理
- 事件窗口/scenarios 作为当前主结果模型
- 独立 rule_specs 作为当前执行链

## 3. 当前阶段范围内必须满足的需求

### 3.1 数据输入与加载

- 支持 CSV、Excel、文本 DAT、MAT、MDF、MF4、BLF 输入。
- 支持在 BLF 与总线类 MDF/MF4 场景中结合 DBC 解码工程信号。
- 支持接口映射 Excel 文件，系统信号由程序自动汇总，用户只维护实际信号名。
- 支持自定义信号表，供曲线信号库与后续扩展使用。

### 3.2 信号映射与标准化

- 接口映射系统表第一列必须自动来自“当前全部 KPI raw_inputs + 当前全部派生量 raw_inputs”的并集。
- 接口映射系统表必须显示 From 列，标识每个 raw_input 当前被哪些 KPI/派生量引用。
- 分析前必须校验系统映射表中每一行是否填写了至少一个实际信号名；未填写时必须实时高亮并阻止分析。
- 当前主映射机制必须是手工映射名逐列匹配，不允许把“自动猜测原始列名”写成主逻辑。

### 3.3 派生量机制

- 派生量必须以独立 Python 文件维护。
- 每个派生量文件必须包含：DERIVED_SIGNAL_DEFINITION、calculate_signal(dataframe)、description、algorithm_summary、CALIBRATION。
- 派生量通常应输出与数据等长、随时间连续变化的过程曲线。
- 若派生量表达的是“整个文件的固有标量属性”，允许输出全时段同一数值的水平直线；但模板和说明中必须明确要求先与用户确认，未确认前不允许擅自采用这种语义。
- 多个 KPI 共用的中间量必须优先抽为派生量。

### 3.4 KPI 机制

- KPI 必须以独立 Python 文件维护。
- 每个正式 KPI 文件必须包含：KPI_DEFINITION、CALIBRATION、calculate_kpi(dataframe)、calculate_kpi_series(dataframe)。
- calculate_kpi_series 是硬性需求，不是可选项。
- KPI_DEFINITION["trend_source"] 必须与 KPI_DEFINITION["name"] 完全一致。
- KPI 的判定口径必须由 threshold、pass_condition、rule_description、pass_message、fail_message 等字段显式描述。
- 可调算法参数必须集中写在 CALIBRATION 中，不允许零散分布在文件顶部和函数内部。

### 3.5 GUI 工作流

- 支持文件入队、分组选择、批量分析。
- 支持同一路径文件按不同 KPI 分组重复分析。
- 结果页必须按“分组标题 -> 文件标题 -> KPI 行”展示。
- 曲线页必须支持：
  - 独立文件选择
  - 多面板显示
  - 横轴联动缩放
  - 多信号拖拽
  - 子框之间拖拽转移
  - 子框配置自动记忆
- 曲线信号库必须拆分为接口信号、KPI 信号、派生量信号、自定义信号。
- 运行日志面板必须主要承载：分析结果、导出文件、报错跳转、KPI/派生量中的 print 输出；不得被无关操作噪音淹没。
- 每次点击分析时，运行日志必须自动清空。
- KPI/派生量/模板编辑器必须支持 Ctrl+S 保存和 Ctrl+F 查找替换。
- KPI 编辑器必须支持双击 derived_inputs 跳转到对应派生量。
- KPI 分组页必须支持双击 KPI 名称跳转到 KPI 编辑器。
- 从接口映射表跳转到 KPI/派生量定义时，必须定位并高亮对应 raw_inputs 项。
- KPI 编辑器至少要持续高亮：raw_inputs、derived_inputs、threshold、pass_condition、CALIBRATION。
- 派生量编辑器至少要持续高亮：raw_inputs、derived_inputs、CALIBRATION。

### 3.6 报告导出

- 面向用户的导出格式仅保留 HTML 和 Word。
- HTML 和 Word 的内容结构必须对齐。
- 报告结构必须与结果页一致，至少包含：分组标题、文件标题、KPI 名称、说明、单位、数值、达标要求、结果。
- 报告模板必须可在 GUI 中直接编辑。
- 报告模板示例必须足以让 AI 根据本地 Word 版式要求翻译生成 HTML 模板。

### 3.7 文档与治理

- 仓库内必须存在完整的需求、架构、实现逻辑、入口、状态、待办、验证、规则、决策、路线、交接文档矩阵。
- 文档必须足以支持陌生接手者快速建立当前心智模型并继续开发。
- 每次开发结束后必须同步更新文档，不允许把关键信息只留在会话里。

## 4. 当前明确不在范围内

- 图形化规则编排器
- 人工复核和备注回填闭环
- 云端服务或联网协同
- 黑盒式 AI 自动最终判定
- 自动标定回写

## 5. 当前验收口径

### 5.1 功能验收

- GUI 能完成主流程：入队、分析、结果查看、曲线查看、配置编辑、导出。
- CLI 能完成单文件或目录批量分析。
- HTML 和 Word 汇总报告均可生成。

### 5.2 逻辑验收

- 接口映射只以 raw_inputs + 手工实际信号名为用户可见输入模型。
- KPI 过程曲线必须可查看。
- 派生量输出语义必须清晰，并由模板约束 AI 生成内容。

### 5.3 文档验收

- 新接手者只阅读仓库文档即可完成环境搭建、现状理解和后续接管。
- 文档中的当前状态、当前阻塞、当前验证结果必须与代码和测试事实一致。
